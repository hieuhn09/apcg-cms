Tổng quan Kiến trúc

Hệ thống APC Central CMS vận hành theo mô hình Multi-Tenant (Đa khách thuê) sử dụng Payload CMS (v3) làm lõi quản trị dữ liệu cấu hình chung. Tất cả các website tin tức thành viên (ví dụ: brief-asia, dtw-web, v.v.) đều kết nối chung vào một cơ sở dữ liệu Postgres duy nhất.

Nguyên tắc cốt lõi:

Mô hình chia sẻ chung Database schema thay vì tách riêng DB vật lý cho mỗi website giúp tối ưu hóa tài nguyên máy chủ, đồng bộ mã nguồn thống nhất và giảm thiểu chi phí quản lý vận hành khi quy mô số lượng website tăng lên hàng chục hoặc hàng trăm.

Phân loại Schemas & Bảng

Để đảm bảo hiệu năng và tính cô lập dữ liệu, các bảng trong cơ sở dữ liệu được chia làm 3 nhóm chính dưới đây:

Nhóm các bảng lưu trữ thông tin hệ thống chung, không trực thuộc bất kỳ một website đơn lẻ nào. Không có trường liên kết tenant trong dữ liệu.

👤
users
Global Identity
Thông tin định danh quản trị viên toàn hệ thống và ánh xạ quyền (roles) tương ứng trên từng website/tenant riêng biệt.
🌐
countries
Global ISO Reference
Bảng tham chiếu quốc gia (ISO reference data), chia sẻ chung danh sách mã quốc gia thống nhất toàn cầu cho tất cả các bài viết.
⚙️
content-engines
Engine Registry
Danh sách các "động cơ" tạo bài viết tự động (Content Engine) được phép đẩy nội dung thô (intake drafts) kèm theo các token định danh đã băm.
📜
activityLog
System Logs
Nhật ký hoạt động chung lưu trữ các sự kiện chỉnh sửa, đăng bài và thao tác quan trọng hỗ trợ điều tra lỗi (audit trail).

Nhóm các bảng chứa thông tin nhận diện và tùy biến nghiệp vụ cho từng website. Đây là cầu nối trung gian xác định đặc tính hoạt động của từng tenant.

🏢
tenants
Tenant Registry
Mỗi bản ghi tương ứng với một website (ví dụ: brief-asia). Lưu tên, slug, tên miền (domain), các ngôn ngữ kích hoạt và mảng bật/tắt tính năng (features).
🗺️
menus
Navigation Config
Lưu cấu hình menu Header và Footer độc lập cho mỗi trang web. Bản ghi được liên kết trực tiếp với một tenant_id.

Nhóm các bảng chứa nội dung biên tập chính của từng trang web. Các bảng này có trường khóa ngoại tenant để tự động phân tách và cô lập dữ liệu.

📄
articles
Tenant Scoped
Nội dung bài viết, trạng thái xuất bản, lịch đăng, nội dung đã dịch và lịch sử phiên bản của riêng từng website.
✍️
authors
Tenant Scoped
Thông tin hồ sơ của các tác giả/biên tập viên trực thuộc trang web thành viên đó.
🖼️
media
Tenant Scoped
Hình ảnh/video được tải lên thuộc sở hữu của website đó. Tệp tin được upload lên thư mục riêng trong Cloudflare R2 bucket.
🏷️
pillars / sectors / tags
Tenant Scoped
Hệ thống chuyên mục chính (pillars), lĩnh vực (sectors) và thẻ từ khóa phân loại tin tức hoàn toàn khác biệt giữa các trang web.

Chi tiết 22 Schemas hệ thống

Hệ thống APC Central CMS hiện đang có 22 schemas (collections) được cấu hình. Mỗi schema chịu trách nhiệm lưu trữ các phần dữ liệu chuyên biệt sau:

I. Nhóm Hệ thống & Toàn cục (5 Schemas) +
Schema (Slug) Thông tin lưu trữ chính
users
  • Định danh người dùng: email, password (đã mã hóa).
  • Quyền hệ thống cao nhất: role (Standard hoặc Admin).
  • Phân quyền theo tenant: danh sách tenants (liên kết đến tenant, roles như websiteAdmin, editor, contributor, và cờ canPublish).
countries
  • Dữ liệu ISO của các quốc gia: code (Mã quốc gia, ví dụ: VN, SG), name (Tên quốc gia).
  • Sử dụng như danh mục tham chiếu toàn cầu cho các bài viết.
content-engines
  • Danh sách Content Engine được tích hợp: name, tokenHash (mã token băm để xác thực intake).
  • Quyền hạn: danh sách allowedTenants (danh sách các web engine được quyền đẩy bài viết sang), allowedActions (các hành động được phép như create, update).
  • Giám sát: status (active, suspended), rateLimit, và lastSeen.
activityLog
  • Nhật ký thao tác append-only: user (thực hiện), tenant (nếu thuộc phạm vi tenant cụ thể), action (create, update, delete, publish).
  • Chi tiết: collectionName, documentId, changes (dữ liệu trước/sau khi thay đổi).
media
  • Quản lý tập tin đa phương tiện: filename, mimeType, filesize.
  • Dữ liệu ảnh: metadata kích thước (thumbnail, card, hero, v.v.), URL trỏ đến Cloudflare R2 bucket.
  • Dữ liệu SEO/Bản dịch: alt, caption (được cấu hình bản dịch đa ngôn ngữ).
II. Nhóm Cấu hình Website (2 Schemas) +
Schema (Slug) Thông tin lưu trữ chính
tenants
  • Định danh thương hiệu website: name, slug (ví dụ: brief-asia), domain, additionalDomains (bí danh tên miền), frontendUrl (để nhận webhook revalidate).
  • Bản dịch & Múi giờ: defaultLanguage, supportedLanguages (danh sách ngôn ngữ kích hoạt), timezone.
  • Module kích hoạt: danh sách cờ bật/tắt features (newsletters, podcasts, marketData, v.v.).
  • Bảo mật: readTokens (lưu hash của token phía frontend dùng để đọc API public).
menus
  • Khóa ngoại liên kết: tenant.
  • Bố cục định hướng: location (Header, Footer).
  • Cấu trúc liên kết: danh sách các phần tử menu gồm label (tiêu đề hiển thị), url (đường dẫn), và thứ tự hiển thị order.
III. Nhóm Nội dung Tin bài & Phân loại (5 Schemas) +
Schema (Slug) Thông tin lưu trữ chính
articles
  • Khóa ngoại liên kết: tenant (Bắt buộc).
  • Thông tin cốt lõi: title, slug, body (nội dung dạng rich text/lexical), summary (dek).
  • Trạng thái & Lịch trình: workflowStatus (draft, pending_review, approved, scheduled, published, archived), publishedAt.
  • Quan hệ phân loại: pillar, sectors, tags, author, coAuthors, country, countries, heroImage.
  • Theo dõi nguồn & Đồng bộ: origin (engine/manual), lockedFields (các trường không cho phép Content Engine ghi đè), engineDraftId.
  • Bản dịch: sourceLanguage, translationStatus (trạng thái dịch của từng ngôn ngữ con).
authors
  • Khóa ngoại liên kết: tenant.
  • Thông tin tác giả: name, slug, bio (tiểu sử), profileImage (ảnh chân dung liên kết đến media).
  • Liên kết MXH: twitter, linkedin, email.
pillars
  • Khóa ngoại liên kết: tenant.
  • Danh mục lớn của website: name, slug (ví dụ: Business, Politics), description, order (thứ tự sắp xếp).
sectors
  • Khóa ngoại liên kết: tenant.
  • Danh mục ngành/lĩnh vực: name, slug (ví dụ: Technology, Energy), description.
tags
  • Khóa ngoại liên kết: tenant.
  • Thẻ gắn bài viết: name, slug. Được tự động tạo hoặc gán bởi biên tập viên.
IV. Nhóm Module Tính năng Gated (10 Schemas) +

Đây là các bảng được kích hoạt tùy biến dựa theo cấu hình tính năng (Features) của mỗi website. Mọi bảng đều chứa cột liên kết tenant.

Schema (Slug) Thông tin lưu trữ chính
newsletters Lưu bản tin điện tử: subject (tiêu đề mail), body (nội dung mail HTML), scheduledAt (lịch gửi), status (draft, sent, failed). Kích hoạt bằng cờ newsletters.
podcasts Lưu trữ dữ liệu âm thanh: title, description, audioUrl (link file mp3), duration (thời lượng giây), publishDate, và các tập tin phụ đề đi kèm. Kích hoạt bằng cờ podcasts.
marketSnapshots Dữ liệu thị trường tài chính nhanh hiển thị lên widget: symbol (mã cổ phiếu/tài sản), price, changePercentage (tỷ lệ tăng giảm), lastUpdated. Kích hoạt bằng cờ marketData.
fxRates Tỷ giá ngoại tệ trực tiếp: baseCurrency (ví dụ: USD), targetCurrency (ví dụ: VND), rate (tỷ giá thực tế), timestamp. Kích hoạt bằng cờ marketData.
trendingBlocks Khối chứa danh sách mã/tài sản đang thịnh hành được ghim trên trang chủ. Kích hoạt bằng cờ marketData.
sponsorSlots Quản lý vị trí quảng cáo tài trợ: sponsorName, imageUrl, clickUrl, placement (ví dụ: sidebar, top_banner), startDate, endDate để tự động hạ banner khi hết hạn quảng cáo. Kích hoạt bằng cờ sponsorSlots.
wireDrops Nguồn cấp tin nhanh/tin vắn (wire news): title, content (tin ngắn gọn), source, publishedAt. Kích hoạt bằng cờ wireDrops.
corrections Ghi nhận đính chính sửa lỗi sai bài viết: liên kết article bị sửa đổi, originalText (văn bản gốc bị sai), correctedText (văn bản sửa lại), reason, correctedAt. Kích hoạt bằng cờ corrections.
translationJobs Hàng đợi dịch thuật tự động: liên kết article cần dịch, targetLanguage, status (pending, processing, completed, failed), engineUsed (ví dụ: Google Translate, DeepL), rawResponse. Kích hoạt bằng cờ translations.
engineConflictLog Nhật ký xử lý xung đột nội dung: ghi lại vết khi Content Engine đẩy bài viết mới nhưng bị hệ thống CMS từ chối ghi đè do biên tập viên con người đã khóa trường (lock fields) hoặc tự chỉnh sửa trước đó. Lưu article, fieldName, deniedValue, và reason.

Chi tiết các loại mối quan hệ nghiệp vụ chính:

Mối quan hệ 1-N giữa Tenant và Dữ liệu biên tập (Scoped Content) +

Một Tenant (Website tin tức) sở hữu nhiều bài viết, tác giả, hình ảnh, chuyên mục và cấu hình điều hướng:

  • Foreign Key: Tất cả bảng dữ liệu biên tập đều bắt buộc có cột tenant (kiểu liên kết relationship trỏ đến bảng tenants).
  • Cách hoạt động: Khi một truy vấn lấy danh sách bài viết từ phía Web, hệ thống luôn thêm điều kiện where tenant = [tenant_id] để phân mảnh dữ liệu.
  • Các thực thể áp dụng: articles, authors, media, tags, pillars, sectors, menus, podcasts, newsletters.
Mối quan hệ N-N giữa User và Tenant (Phân quyền chi tiết) +

Một tài khoản người dùng quản trị có thể có các vai trò quản lý khác nhau trên nhiều website khác nhau:

  • Cơ chế lưu trữ: Trong Payload CMS, mối quan hệ này được chuẩn hóa dưới dạng một mảng các đối tượng (Array field) tenants nằm ngay trong bảng users.
  • Cấu trúc bản ghi: Mỗi dòng trong mảng chứa { tenant: TenantID, roles: ['websiteAdmin', 'editor'], canPublish: true }.
  • Cách hoạt động: Khi User truy cập giao diện quản trị, Payload plugin sẽ đọc mảng này để lọc danh sách các Tenant mà User có quyền chuyển đổi qua lại, đồng thời khóa quyền chỉnh sửa các bài viết thuộc Tenant mà User không có quyền truy cập.
Mối quan hệ giữa Bài viết (Articles) và các thực thể khác +

Bảng articles là bảng trung tâm về mặt nội dung và có các quan hệ liên kết phức tạp:

Trường (Field) Kiểu liên kết Bảng đích Ý nghĩa
tenant 1 - 1 (Bắt buộc) tenants Xác định bài viết thuộc trang web nào.
pillar Many-to-One (Bắt buộc) pillars Chuyên mục chính của bài viết (ví dụ: Economy, Tech).
author Many-to-One (Bắt buộc) authors Tác giả biên tập chính.
coAuthors Many-to-Many (Tùy chọn) authors Các đồng tác giả cùng tham gia viết bài.
tags Many-to-Many tags Danh sách các từ khóa phân loại bài viết.

heroImage Many-to-One media Hình ảnh bìa của bài viết, lưu trữ tập tin trong R2.

Chiến Lược Cô Lập Dữ Liệu

Bảo mật dữ liệu chéo giữa các website thành viên là yêu cầu sống còn. APC Central CMS thực thi nguyên tắc bảo mật thông qua 3 lớp phòng vệ:

1. Lớp Ứng Dụng: Payload Multi-Tenant Plugin & Access Controls +

Payload CMS tự động tiêm giá trị tenant vào mọi bộ lọc truy vấn đối với người dùng đang đăng nhập trong trang quản trị Admin. Người dùng của website A không thể xem, sửa hoặc xóa dữ liệu của website B vì hook phân quyền luôn chèn điều kiện:

// access/helpers.ts - Chỉ cho phép đọc tài liệu thuộc tenant của user
export const isAdminOrHasTenantAccess = (collectionName: string) => {
  return ({ req: { user } }) => {
    if (!user) return false;
    if (user.role === 'admin') return true;
    
    // Tạo filter truy vấn ép buộc lấy đúng tenant của user
    const tenantIds = user.tenants?.map(t => t.tenant.id) || [];
    return {
      tenant: {
        in: tenantIds
      }
    };
  };
};
2. Lớp API: Scoped Database Helper (Trực quan & Bắt buộc) +

Do luồng API máy đọc (Engine Intake, Public API) không qua cơ chế xác thực User của Payload, để tránh rò rỉ dữ liệu, lập trình viên không được dùng trực tiếp payload.find() mà phải dùng helper scopedFind():

// src/lib/scoped.ts - Đảm bảo luôn lọc theo tenant_id của request
export async function scopedFind(payload, collection, tenantId, query = {}) {
  return await payload.find({
    collection,
    where: {
      and: [
        { tenant: { equals: tenantId } },
        query
      ]
    }
  });
}
3. Lớp Database: Composite Unique Indexes (Chỉ mục hỗn hợp) +

Để cho phép các website khác nhau có thể sử dụng trùng đường dẫn (ví dụ: cả brief-asia và dtw-web đều có chuyên mục /tag/technology), trường slug không thể cấu hình là UNIQUE toàn cầu.

Hệ thống định nghĩa ràng buộc duy nhất phức hợp trên cơ sở dữ liệu:

CREATE UNIQUE INDEX articles_tenant_slug_idx ON articles (tenant_id, slug);

Điều này cho phép cùng một slug tồn tại trên các website khác nhau, nhưng trên một website thì slug bắt buộc phải là duy nhất.

Tránh việc tạo ra bảng brief_asia_podcasts hay dtw_podcasts. Việc tạo bảng riêng theo tên website dẫn đến phân mảnh mã nguồn, không thể tái sử dụng tính năng và làm phức tạp hóa quá trình nâng cấp cơ sở dữ liệu.

Phương án 1: Chia sẻ Bảng (Shared Table) + Feature Flag (Khuyên dùng) +

Tạo một bảng chung duy nhất tên là podcasts trong cơ sở dữ liệu và liên kết với trường tenant. Tính năng này sẽ được quản lý bằng bộ cờ Features cấu hình trong bản ghi tenants.

1
Thêm trường Feature Flag vào Tenants Collection
Định nghĩa trường checkbox podcasts bên trong danh sách features trong cấu hình tenant.
2
Áp dụng Access Control ẩn/hiện ở Admin UI
Viết hook kiểm tra quyền truy cập cho Collection Podcasts. Nếu tenant hiện tại không kích hoạt tính năng này, bộ sưu tập sẽ ẩn khỏi thanh điều hướng Admin của người dùng đó.
3
Chặn kiểm soát ở mức API
Khi client gọi API đọc/ghi podcasts, route handler kiểm tra xem tenant của client đó có bật feature podcasts hay không. Nếu không, trả về lỗi 404 Not Found hoặc 422 Unprocessable Entity.

Ưu điểm: Database sạch sẽ, dễ bảo trì. Nếu trong tương lai Web B muốn dùng lại tính năng Podcast, admin chỉ cần tích chọn bật tính năng trên giao diện Tenants mà không cần thay đổi bất cứ dòng code hay chạy Migration DB nào.

Phương án 2: Tạo Table riêng chỉ dành cho một Web cụ thể (Bespoke Table) +

Nếu tính năng là cực kỳ cá biệt, chỉ phục vụ duy nhất một website A và không thể dùng chung cho bất kỳ trang nào khác, ta thiết lập một Collection mới nhưng giới hạn cứng quyền truy cập ở mức ứng dụng.

// collections/BespokeFeature.ts
export const BespokeFeature = {
  slug: 'bespoke-feature',
  access: {
    // Chỉ cho phép đọc/ghi nếu tenant ID khớp với ID của Website A cụ thể
    read: ({ req }) => req.user?.tenants?.some(t => t.tenant.slug === 'brief-asia'),
    create: ({ req }) => req.user?.tenants?.some(t => t.tenant.slug === 'brief-asia'),
  },
  fields: [
    {
      name: 'tenant',
      type: 'relationship',
      relationTo: 'tenants',
      required: true,
      // Ẩn trường này hoặc khóa mặc định theo tenant của user
    }
  ]
}

Lưu ý: Vẫn bắt buộc phải duy trì cột tenant trên bảng này để giữ tính nhất quán về cơ cấu dữ liệu phân quyền và chỉ mục đa khách thuê.

Phương án 3: Lưu trữ Cơ sở dữ liệu độc lập bên ngoài (External Service) +

Với các dữ liệu phát sinh từ phía người đọc cuối (Reader / End-user) của từng website cụ thể như: tài khoản người đọc, lịch sử thanh toán, lượt thích bài viết, bình luận, v.v.

Cách xử lý: Không đưa các dữ liệu này vào trong hệ thống Central CMS. Hãy triển khai cơ sở dữ liệu riêng tại mã nguồn frontend của website thành viên đó (hoặc một cụm Backend nhỏ chuyên biệt của web đó). Các dữ liệu này chỉ cần tham chiếu đến Central CMS bằng các khóa liên kết có định danh duy nhất không đổi như articleId hoặc articleSlug.

Hướng dẫn triển khai tạo Bảng mới (Podcast Example)

Để thêm bảng podcasts sử dụng Phương án 1 (Khuyên dùng), hãy làm theo quy trình 4 bước:

Bước 1: Khai báo collection trong Payload CMS 
// src/collections/Podcasts.ts
import { CollectionConfig } from 'payload';
import { isAdminOrHasTenantAccess } from '../access/helpers';

export const Podcasts: CollectionConfig = {
  slug: 'podcasts',
  admin: {
    useAsTitle: 'title',
    // Ẩn collection nếu tenant hiện tại của user không bật feature này
    hidden: ({ user, locale }) => {
      // Logic kiểm tra xem tenant của user có bật podcasts hay không
      return !user?.tenants?.some(t => t.tenant.features?.includes('podcasts'));
    }
  },
  access: {
    read: isAdminOrHasTenantAccess('podcasts'),
    create: isAdminOrHasTenantAccess('podcasts'),
    update: isAdminOrHasTenantAccess('podcasts'),
    delete: isAdminOrHasTenantAccess('podcasts'),
  },
  fields: [
    {
      name: 'title',
      type: 'text',
      required: true,
    },
    {
      name: 'audioUrl',
      type: 'text',
      required: true,
    },
    {
      name: 'tenant',
      type: 'relationship',
      relationTo: 'tenants',
      required: true,
      admin: {
        position: 'sidebar',
      }
    }
  ]
};
Bước 2: Cập nhật file cấu hình chính 
// payload.config.ts
import { buildConfig } from 'payload/config';
import { Podcasts } from './src/collections/Podcasts';

export default buildConfig({
